## 1. 本文書の目的

AEGIS-SIGHT に IAMS から選択的に移植する全機能要件を、要件ID・詳細仕様・優先度・受入条件の形式で一覧化する。本文書は開発・テスト・受入検査の基準文書として使用する。

---

## 2. 優先度の定義

| 優先度 | 定義 | 対応方針 |
|--------|------|----------|
| **必須（Must）** | リリース時に実装必須。未実装の場合はリリース不可 | Phase 2-3 で実装完了 |
| **推奨（Should）** | 実装を強く推奨。リソース制約時は Phase 5 以降に延期可 | Phase 3-4 で実装 |
| **任意（Could）** | あれば利便性が向上するが、省略可能 | バックログに登録 |

---

## 3. 機能要件一覧

### 3.1 機能領域 OBS：Prometheus/Grafana インフラ可観測性

IAMS が保有していたインフラ監視基盤を AEGIS-SIGHT に移植し、サーバ・エンドポイント層の可視化を実現する。

| 要件ID | 要件名 | 優先度 | 詳細仕様 | 受入条件 |
|--------|--------|--------|----------|----------|
| OBS-001 | Prometheus メトリクス収集 | 必須 | Windows エージェント（Windows Exporter）からサーバリソース（CPU/メモリ/ディスク/ネットワーク）のメトリクスを Prometheus 形式で収集する。スクレイプ間隔は 60 秒。対象は AEGIS-SIGHT API サーバ、Windows エンドポイント、PostgreSQL。 | (1) Prometheus がスクレイプ対象から 60 秒間隔でメトリクスを取得できること (2) CPU/メモリ/ディスク/ネットワークの各メトリクスが Prometheus UI で確認できること (3) Windows Exporter が 500 台規模のエンドポイントに対応できること |
| OBS-002 | Grafana 統合ダッシュボード | 必須 | インフラ状態・エンドポイント状態を Grafana でリアルタイム可視化する。ダッシュボード構成：AEGIS-SIGHT Overview、Windows Endpoints、SAM License Status、Procurement Pipeline、Security Posture の 5 種類。 | (1) 5 種類のダッシュボードが正常に表示されること (2) 各ダッシュボードがリアルタイム（最大 60 秒遅延）でデータを反映すること (3) Grafana データソースとして Prometheus が正常に接続されていること |
| OBS-003 | アラートルール設定 | 必須 | CPU 90%超過（5分持続）、ディスク残量 10%未満（1分持続）、サービス停止、ライセンス超過の各条件でアラートを発行する。Alertmanager 経由で通知を配信する。 | (1) CPU 90%超過が 5 分持続した場合に warning アラートが発行されること (2) ディスク残量 10%未満で critical アラートが発行されること (3) ライセンス超過検知で即座に critical アラートが発行されること (4) Alertmanager からの通知が正常に配信されること |
| OBS-004 | AEGIS-SIGHT ダッシュボードへの埋め込み | 推奨 | Grafana パネルを AEGIS-SIGHT の Next.js ダッシュボードに iframe 形式で埋め込み、統一的な管理画面を提供する。 | (1) Next.js ダッシュボードに Grafana パネルが正常に表示されること (2) iframe 埋め込みでの認証が SSO と連携すること (3) レスポンシブデザインに対応すること |

### 3.2 機能領域 PWA：PWA 対応（オフライン UI）

建設現場等のネットワーク不安定環境で AEGIS-SIGHT を利用可能にするため、PWA 機能を移植する。

| 要件ID | 要件名 | 優先度 | 詳細仕様 | 受入条件 |
|--------|--------|--------|----------|----------|
| PWA-001 | Service Worker 実装 | 必須 | ネットワーク断絶時にキャッシュされた資産情報を参照可能にする。キャッシュ戦略は Network First（API レスポンス）と Cache First（静的アセット）のハイブリッド方式。キャッシュ名は `aegis-sight-v1`、対象は `/api/v1/assets`、`/api/v1/sam/licenses`、`/offline.html`。 | (1) Service Worker が正常にインストール・アクティベートされること (2) ネットワーク断絶時にキャッシュ済みデータが表示されること (3) ネットワーク正常時は最新データが取得されキャッシュが更新されること |
| PWA-002 | オフライン資産確認機能 | 必須 | 建設現場でネットワーク不通時でも端末の割当・状態を確認できる。直近 30 日分の資産情報をキャッシュ対象とする。 | (1) オフライン状態で資産一覧が表示されること (2) 端末の割当状態、ステータスが確認できること (3) キャッシュされた情報が直近 30 日分であること |
| PWA-003 | オンライン復帰時の自動同期 | 必須 | ネットワーク復帰後にオフライン操作（状態更新等）を Background Sync API を使用して自動同期する。同期タグは `sync-offline-operations`。 | (1) オフライン中の操作がローカルに蓄積されること (2) ネットワーク復帰後に自動的にサーバと同期されること (3) 同期失敗時にリトライが実行されること (4) 同期完了後にユーザへ通知されること |
| PWA-004 | ホーム画面追加対応 | 推奨 | スマートフォン・タブレットのホーム画面に追加可能なマニフェスト設定。アプリ名「AEGIS-SIGHT IT管理」、表示モード standalone、テーマカラー #2563EB。 | (1) manifest.json が正しく配信されること (2) iOS/Android でホーム画面に追加できること (3) standalone モードで起動すること (4) アイコン（192x192、512x512）が正常に表示されること |

### 3.3 機能領域 PRC：調達管理（Procurement）

IT 資産のライフサイクル全体（調達→受領→使用→保守→廃棄）を一貫管理する機能を移植する。

| 要件ID | 要件名 | 優先度 | 詳細仕様 | 受入条件 |
|--------|--------|--------|----------|----------|
| PRC-001 | 調達申請・承認ワークフロー | 必須 | IT 機器・ソフトウェアの調達申請から上長承認・発注までのワークフローを実装する。ステータス遷移：draft → submitted → approved/rejected → ordered。カテゴリ：hardware / software / service / consumable。申請番号は PRC-YYYY-NNNN 形式で自動採番。 | (1) 調達申請が起票・提出できること (2) 承認者が申請を承認/却下できること (3) 承認後に発注ステータスに遷移できること (4) 申請番号が一意に自動採番されること (5) 承認履歴が記録されること |
| PRC-002 | 受領登録・資産台帳連携 | 必須 | 機器受領時に AEGIS-SIGHT 資産台帳（assets テーブル）に自動登録する。ハードウェアの場合はシリアル番号・購入日・購入費用・所属部門を自動転記。調達申請レコードと資産レコードを asset_id で紐付ける。 | (1) 受領登録で資産台帳にレコードが自動作成されること (2) シリアル番号・購入情報が正しく転記されること (3) 調達申請と資産レコードがリンクされていること (4) ソフトウェアの場合はライセンス管理（SAM）と連携すること |
| PRC-003 | 廃棄管理 | 必須 | 廃棄申請・廃棄証明書管理・台帳からの除却記録を管理する。廃棄証明書ファイルは Azure Blob Storage に保存し、7 年間保持する（J-SOX 対応）。ステータス遷移：active → disposal_requested → disposed。 | (1) 廃棄申請が起票・承認できること (2) 廃棄証明書ファイルがアップロード・保存できること (3) 廃棄完了時に資産台帳から除却（ステータス変更）されること (4) 廃棄記録が 7 年間保持されること |
| PRC-004 | 調達ライフサイクル追跡 | 必須 | 調達→受領→使用中→保守→廃棄の全ライフサイクルを一元管理する。各ステータス変更時にタイムスタンプと操作者を記録する。Grafana Procurement Pipeline ダッシュボードで可視化する。 | (1) 全ステータス遷移が正常に動作すること (2) 各遷移のタイムスタンプ・操作者が記録されること (3) ライフサイクル全体がダッシュボードで可視化されること (4) 不正なステータス遷移が拒否されること |
| PRC-005 | 費用管理 | 推奨 | 調達費用・保守費用の記録および部門別コスト集計を行う。単価・数量・合計金額を記録。通貨はデフォルト JPY。費用情報は管理者ロール（admin）のみアクセス可能。 | (1) 費用情報が正しく記録・計算されること (2) 部門別コスト集計レポートが出力できること (3) 管理者ロール以外のユーザは費用情報にアクセスできないこと (4) 月次・四半期・年次での集計が可能なこと |

### 3.4 機能領域 SAM：SAM ライセンス管理

AEGIS-SIGHT の SW インベントリ機能と統合し、ライセンスコンプライアンスを自動管理する。

| 要件ID | 要件名 | 優先度 | 詳細仕様 | 受入条件 |
|--------|--------|--------|----------|----------|
| SAM-001 | ライセンス登録・管理 | 必須 | ソフトウェアライセンスの登録・管理機能。管理項目：ソフトウェア名、ベンダー名、ライセンス種別（perpetual/subscription/oem/volume/freeware/open_source）、ライセンスキー（AES-256-GCM 暗号化）、購入ライセンス数、単価、購入日、有効期限、ベンダー管理番号。 | (1) ライセンス情報の CRUD 操作が正常に動作すること (2) ライセンスキーが暗号化されて保存されること (3) ライセンス種別のバリデーションが正しく動作すること (4) 1,000 件以上のライセンスを管理できること |
| SAM-002 | インベントリとの自動照合 | 必須 | AEGIS-SIGHT の SW インベントリ収集結果と software_licenses テーブルを自動照合する。Celery タスクとして毎日 03:00 に実行。500 件の SW インベントリを 5 秒以内に照合。未登録ソフトウェアの検出も行う。 | (1) 自動照合が毎日 03:00 に実行されること (2) 500 件の SW インベントリが 5 秒以内に照合完了すること (3) installed_count が自動更新されること (4) 未登録ソフトウェアが検出・報告されること |
| SAM-003 | ライセンス超過アラート | 必須 | インストール数がライセンス購入数を超過した場合に自動アラートを発行する。Prometheus メトリクス（aegis_license_installed_count > aegis_license_purchased_count）で監視。超過検知で即座に critical アラートを発行。 | (1) ライセンス超過時にアラートが発行されること (2) アラートにソフトウェア名・超過数が含まれること (3) 管理者にメール/Slack 通知が送信されること (4) compliance_status が OVER に更新されること |
| SAM-004 | ライセンス期限アラート | 必須 | 有効期限 30 日前・7 日前のアラート通知。license_compliance_view の expiry_status で管理。EXPIRING_SOON（30 日以内）/ EXPIRED（期限切れ）の 2 段階。 | (1) 有効期限 30 日前に warning アラートが発行されること (2) 有効期限 7 日前に critical アラートが発行されること (3) 期限切れライセンスが EXPIRED として表示されること (4) 更新対象一覧レポートが出力できること |
| SAM-005 | SAM レポート（J-SOX） | 必須 | IT 全般統制対応のライセンス管理証跡レポートを自動生成する。レポート内容：ライセンス購入数/インストール数/超過状況、照合履歴、是正措置記録。7 年間保存。 | (1) J-SOX 対応レポートが自動生成されること (2) レポートに必要な全項目が含まれること (3) レポートが PDF/CSV 形式で出力できること (4) 監査人がレポートの正当性を検証できること (5) レポートが 7 年間保存されること |
| SAM-006 | Microsoft 365 ライセンス連携 | 推奨 | Microsoft Graph API 経由で M365 ライセンス割当状況を自動取得する。subscribedSkus エンドポイントから skuPartNumber、consumedUnits、prepaidUnits.enabled を取得し、software_licenses テーブルの m365_assigned カラムに反映。 | (1) Graph API から M365 ライセンス情報が取得できること (2) m365_assigned カラムが自動更新されること (3) M365 ライセンスの超過・期限管理が SAM と統合されること (4) API 認証エラー時にリトライ・通知が行われること |

### 3.5 機能領域 TST：テスト資産の pytest 変換活用

IAMS の 1,157 件の Jest テスト資産を分析・変換し、AEGIS-SIGHT の品質保証に活用する。

| 要件ID | 要件名 | 優先度 | 詳細仕様 | 受入条件 |
|--------|--------|--------|----------|----------|
| TST-001 | IAMS テスト資産分析 | 必須 | IAMS の 1,157 件の Jest テストを分析し、AEGIS-SIGHT に移植すべきテストケースを選定する。分類：Jest 単体テスト（フロント）約 400 件 → Vitest 変換、Jest 統合テスト（API）約 500 件 → pytest + httpx 変換、Jest E2E テスト（Playwright）約 200 件 → 流用、廃棄対象約 57 件（CMDB/インシデント/変更管理関連）。 | (1) 全 1,157 件のテストが分類・判定されていること (2) 移植対象/非対象の判定根拠が文書化されていること (3) 廃棄対象テスト（約 57 件）の廃棄理由が明記されていること |
| TST-002 | pytest 変換・統合 | 必須 | 選定したテストケースを pytest 形式に変換し、AEGIS-SIGHT のテストスイートに統合する。API テストは pytest + httpx、フロントエンドテストは Vitest、E2E テストは Playwright を使用。GIVEN/WHEN/THEN パターンで記述。 | (1) 変換対象テストが全て pytest/Vitest/Playwright 形式で実行可能なこと (2) CI パイプラインで自動実行されること (3) テストが GIVEN/WHEN/THEN パターンで可読性が確保されていること (4) 変換前後でテストの検証内容が同等であること |
| TST-003 | カバレッジ目標達成 | 必須 | テスト変換により AEGIS-SIGHT のカバレッジを 80%以上に引き上げる。カバレッジ測定はバックエンド pytest-cov、フロントエンド Vitest coverage で実施。 | (1) AEGIS-SIGHT 全体のテストカバレッジが 80%以上であること (2) 各モジュール（SAM/調達/OBS/PWA）のカバレッジが 70%以上であること (3) カバレッジレポートが CI で自動生成されること |

## 4. 機能間依存関係

本節では各機能要件間の依存関係を明示する。実装順序の決定に使用する。

| OBS-004 | OBS-002 | Grafana パネルを Next.js に埋め込み |
| PWA-002 | PWA-001 | Service Worker のキャッシュ機能に依存 |
| PWA-003 | PWA-001 | Background Sync API は Service Worker 上で動作 |
| TST-002 | TST-001 | 分析結果に基づいて変換対象を決定 |
| TST-003 | TST-002 | 変換完了後にカバレッジを計測 |
