## 1. 本文書の目的

IAMS リポジトリが保有する全機能を対象に、AEGIS-SIGHT への移植対象/非対象を判定し、その根拠を文書化する。また、移植対象機能と AEGIS-SIGHT 既存機能との統合マッピングを定義する。

本分析は Project_Repository_List_v2_1 の評価結果（統合スコア 78/100）に基づく。

## 4. 移植対象機能の詳細判定

### 4.1 Prometheus/Grafana インフラ可観測性

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植対象** |
| **判定理由** | AEGIS-SIGHT はエンドポイント管理に特化しており、サーバ・インフラ層の可観測性が不足している。IAMS の Prometheus/Grafana スタックを移植することで、エンドポイントからインフラまでの一元監視を実現する。 |
| **AEGIS-SIGHT での価値** | (1) サーバリソース（CPU/メモリ/ディスク/ネットワーク）のリアルタイム監視 (2) アラート基盤の統一 (3) 統合ダッシュボードによる全体可視化 |
| **技術的実現性** | 高。Prometheus/Grafana は Docker コンテナとして独立デプロイ可能。AEGIS-SIGHT API への metrics エンドポイント追加のみで統合可能。 |
| **移植対象コンポーネント** | prometheus.yml（スクレイプ設定）、Grafana ダッシュボード JSON 5 種類、アラートルール YAML、Docker Compose サービス定義 |
| **移植工数見積り** | 2 週間 |

### 4.2 PWA 対応（オフライン UI）

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植対象** |
| **判定理由** | みらい建設工業の業務特性上、建設現場でのネットワーク不安定環境でのシステム利用が必須要件。IAMS の PWA 実装を移植し、AEGIS-SIGHT 既存のオフラインバッファ設計と統合する。 |
| **AEGIS-SIGHT での価値** | (1) 建設現場での資産確認・ステータス更新 (2) ネットワーク復帰時の自動データ同期 (3) モバイルデバイスからのアクセス改善 |
| **技術的実現性** | 高。AEGIS-SIGHT は Next.js 14 を採用しており、Service Worker の追加は比較的容易。既存のオフラインバッファ設計との統合が必要。 |
| **移植対象コンポーネント** | service-worker.ts、manifest.json、オフラインストレージ管理ロジック、Background Sync 実装 |
| **移植工数見積り** | 1.5 週間 |
| **既存機能との統合ポイント** | AEGIS-SIGHT の「現場 PC オフラインバッファ対応」機能に PWA レイヤを追加する形で統合 |

### 4.3 調達管理（Procurement）

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植対象** |
| **判定理由** | AEGIS-SIGHT は IT 資産の「管理」に特化しているが、「取得（調達）」から「廃棄」までのライフサイクル全体をカバーしていない。IAMS の調達管理を移植することで、購入から廃棄までの一貫管理を実現する。 |
| **AEGIS-SIGHT での価値** | (1) 調達→資産台帳自動登録のシームレスな連携 (2) 承認ワークフローによるガバナンス強化 (3) 廃棄証跡の管理（J-SOX 対応） (4) 部門別コスト管理 |
| **技術的実現性** | 中〜高。FastAPI でのワークフロー API 実装、Next.js での承認 UI 実装が必要。AEGIS-SIGHT の既存 assets テーブルとの連携（asset_id による紐付け）が核心。 |
| **移植対象コンポーネント** | procurement_requests テーブル定義、ProcurementService（ワークフローロジック）、調達管理 UI（申請/承認/受領/廃棄）、Grafana Procurement Pipeline ダッシュボード |
| **移植工数見積り** | 3 週間 |
| **既存機能との統合ポイント** | AEGIS-SIGHT 資産台帳（assets テーブル）との双方向連携。受領時に資産レコード自動作成、廃棄時に資産ステータス自動更新。 |

### 4.4 SAM ライセンス管理

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植対象** |
| **判定理由** | AEGIS-SIGHT は SW インベントリ収集機能を保有しているが、収集結果をライセンスと照合する機能がない。IAMS の SAM 機能を統合することで、ライセンスコンプライアンスの自動管理を実現する。ISO27001 A.5.9（資産目録）および J-SOX IT 全般統制の要件充足にも直結する。 |
| **AEGIS-SIGHT での価値** | (1) ライセンス超過の自動検知（コンプライアンスリスク軽減） (2) J-SOX 監査対応レポート自動生成 (3) M365 ライセンスの自動管理 (4) ライセンスコスト最適化 |
| **技術的実現性** | 高。AEGIS-SIGHT の SW インベントリデータを入力として、SAM 照合エンジンを Celery タスクとして実装。既存の Graph API 連携基盤を M365 ライセンス取得に流用可能。 |
| **移植対象コンポーネント** | software_licenses テーブル、license_compliance_view、SAMService（照合エンジン）、SAM 管理 UI（ライセンス一覧/照合結果/レポート）、M365 連携ロジック |
| **移植工数見積り** | 3 週間 |
| **既存機能との統合ポイント** | AEGIS-SIGHT SW インベントリ収集結果を照合入力として直接利用。Prometheus メトリクスでライセンス超過を監視。 |

### 4.5 テスト資産（1,157 件）

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植対象（変換活用）** |
| **判定理由** | IAMS の Jest テスト 1,157 件は、移植機能の品質保証に直接活用可能。特に SAM 照合・調達ワークフローのテストケースは、AEGIS-SIGHT でのテスト設計工数を大幅に削減できる。 |
| **変換方針** | 下表参照 |
| **移植工数見積り** | 2 週間（変換+統合） |

#### テスト変換マッピング

| IAMS テスト種別 | 推定件数 | 変換先 | 変換方針 |
|--------------|----------|--------|---------|
| Jest 単体テスト（フロントエンド） | 約 400 件 | Vitest | Next.js コンポーネントテストとして移植。React Testing Library は共通利用。 |
| Jest 統合テスト（API） | 約 500 件 | pytest + httpx | FastAPI テストクライアント（httpx.AsyncClient）で実装。GIVEN/WHEN/THEN パターン。 |
| Jest E2E テスト（Playwright） | 約 200 件 | Playwright（そのまま流用） | Playwright は IAMS/AEGIS-SIGHT 共通のため、URL・セレクタの修正のみ。 |
| 廃棄対象テスト | 約 57 件 | 廃棄 | CMDB/インシデント/SLA/変更管理関連のテスト。移植非対象機能のため廃棄。 |

## 5. 移植非対象機能の詳細判定

### 5.1 CMDB（構成管理データベース）

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植非対象（廃棄）** |
| **廃棄理由** | AEGIS-SIGHT 本体の資産管理 DB（assets テーブル群）と**完全に機能が重複**する。AEGIS-SIGHT の資産管理機能は SKYSEA Client View 同等の詳細な HW/SW 情報を収集しており、IAMS CMDB は劣後する。 |
| **重複する AEGIS-SIGHT 機能** | IT 資産管理（HW 情報収集）、SW インベントリ管理、資産台帳管理 |
| **廃棄による影響** | なし。AEGIS-SIGHT 資産管理が上位互換として全機能をカバー。 |
| **廃棄対象テスト** | 約 150 件（CMDB 関連テスト） |

### 5.2 インシデント管理

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植非対象（廃棄）** |
| **廃棄理由** | ITSM-System（別リポジトリ）と**完全に機能が重複**する。みらい建設工業のインシデント管理は ITSM-System で運用されており、IAMS のインシデント管理を AEGIS-SIGHT に移植する必要はない。 |
| **重複するシステム** | ITSM-System（Kensan196948G/ITSM-System） |
| **廃棄による影響** | なし。ITSM-System が正式なインシデント管理基盤。 |
| **廃棄対象テスト** | 約 120 件（インシデント管理関連テスト） |

### 5.3 SLA 管理

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植非対象（廃棄）** |
| **廃棄理由** | ITSM-System と**完全に機能が重複**する。SLA の定義・監視・報告は ITSM-System の責務であり、AEGIS-SIGHT はエンドポイント管理に集中する。 |
| **重複するシステム** | ITSM-System（Kensan196948G/ITSM-System） |
| **廃棄による影響** | なし。SLA 管理は ITSM-System の管轄。 |
| **廃棄対象テスト** | 約 80 件（SLA 管理関連テスト） |

### 5.4 変更管理

| 判定項目 | 内容 |
|---------|------|
| **判定** | **移植非対象（廃棄）** |
| **廃棄理由** | ITSM-System（統合プラン A）と**重複**する。変更管理プロセスは ITSM-System に集約する方針。 |
| **重複するシステム** | ITSM-System（統合プラン A で変更管理を含む） |

## 6. 既存機能との統合マッピング

### 6.1 統合マッピング概要

| # | AEGIS-SIGHT 既存機能 | IAMS 移植機能 | 統合方式 | 統合ポイント |
|---|---------------------|-------------|---------|-------------|
| 1 | IT 資産管理（HW 情報収集） | 調達管理（PRC） | **ライフサイクル統合** | 受領登録時に assets テーブルへ自動レコード作成。procurement_requests.asset_id で双方向リンク。 |
| 2 | SW インベントリ管理 | SAM ライセンス管理（SAM） | **照合統合** | SW インベントリ収集結果を SAMService の入力として利用。installed_count を自動更新。 |
| 3 | 統合ダッシュボード（Next.js） | Grafana インフラ監視（OBS） | **表示統合** | Grafana パネルを Next.js ダッシュボードに iframe 埋め込み。認証は SSO 連携。 |
| 4 | 現場 PC（オフラインバッファ対応） | PWA オフライン UI（PWA） | **レイヤ統合** | 既存オフラインバッファ設計に Service Worker レイヤを追加。データ同期ロジックを統合。 |

### 6.2 統合ポイント詳細

#### 統合ポイント 1：調達管理 → 資産台帳

```
[調達申請]                    [AEGIS-SIGHT 資産台帳]
procurement_requests          assets
  ├── item_name         →     ├── asset_name
  ├── category          →     ├── asset_type
  ├── total_price       →     ├── purchase_cost
  ├── department_id     →     ├── department_id
  ├── received_at       →     ├── purchase_date
  └── asset_id  ───────────→  └── id（外部キー連携）
```

- 受領登録（PRC-002）実行時に ProcurementService が AssetService.create() を呼び出し、資産レコードを自動生成
- 廃棄管理（PRC-003）実行時に資産ステータスを disposed に自動更新

#### 統合ポイント 2：SW インベントリ → SAM 照合

```
[AEGIS-SIGHT SW インベントリ]     [SAM ライセンス管理]
sw_inventory_results              software_licenses
  ├── software_name        ←→     ├── software_name（照合キー）
  ├── installed_count      →      ├── installed_count（自動更新）
  └── endpoint_list               ├── purchased_count
                                  └── compliance_status（自動判定）
```

- 日次バッチ（03:00）で SAMService.run_compliance_check() が SW インベントリを集計し、ライセンス登録と照合
- 超過検知時に Prometheus メトリクス更新 + アラート発行

#### 統合ポイント 3：Next.js ダッシュボード → Grafana

```
[Next.js ダッシュボード]          [Grafana]
aegis-sight-web/                  aegis-sight-infra/observability/grafana/
  ├── app/dashboard/              ├── dashboards/
  │   ├── page.tsx                │   ├── overview.json
  │   └── components/             │   ├── windows-endpoints.json
  │       └── GrafanaPanel.tsx    │   ├── sam-license.json
  │           └── <iframe>  ────→ │   ├── procurement-pipeline.json
  │                               │   └── security-posture.json
  └── middleware.ts（SSO）        └── datasources/prometheus.yml
```

- GrafanaPanel コンポーネントが iframe で Grafana パネルを埋め込み
- Grafana の認証は AEGIS-SIGHT の SSO トークンを Proxy Auth で連携

#### 統合ポイント 4：オフラインバッファ → PWA

```
[既存オフラインバッファ]          [PWA レイヤ（追加）]
  ├── ローカルDB（IndexedDB）←── Service Worker キャッシュ
  ├── 差分同期ロジック      ←── Background Sync API
  └── 通信状態監視          ←── navigator.onLine + SW fetch イベント
```

- 既存のオフラインバッファ設計を Service Worker のキャッシュ戦略で拡張
- Background Sync API で既存の差分同期ロジックを置換・強化

## 7. 移植リスクと対策

| # | リスク | 影響度 | 発生確率 | 対策 |
|---|--------|--------|----------|------|
| 1 | IAMS と AEGIS-SIGHT のデータモデル不整合 | 高 | 中 | Phase 1（分析・設計）でデータモデルの詳細比較を実施。マッピング定義書を作成。 |
| 2 | Jest → pytest 変換時のテストロジック欠損 | 中 | 中 | 変換後のテストを手動レビュー。元テストとの同等性を検証チェックリストで確認。 |
| 3 | PWA キャッシュと既存オフラインバッファの競合 | 中 | 低 | 統合設計で責務分担を明確化。Service Worker はネットワーク層、既存バッファはデータ層として分離。 |
| 4 | Prometheus スクレイプ負荷による性能劣化 | 低 | 低 | スクレイプ間隔 60 秒で開始し、負荷状況に応じて調整。file_sd_configs で段階的にターゲット追加。 |
| 5 | M365 Graph API の認証・権限設定の複雑さ | 中 | 中 | Azure AD アプリ登録・権限設定の事前検証。SAM-006 は推奨要件のため、遅延時は Phase 5 以降に延期。 |
