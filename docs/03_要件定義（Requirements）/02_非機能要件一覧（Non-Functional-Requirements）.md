## 1. 本文書の目的

AEGIS-SIGHT に IAMS 機能を統合するにあたり、性能・可用性・セキュリティ・スケーラビリティ・保守性の各観点から非機能要件を定義する。本文書は設計・構築・運用の品質基準として使用する。

---

## 2. 性能要件

### 2.1 応答性能

| 要件ID | 要件名 | 基準値 | 測定方法 | 備考 |
|--------|--------|--------|----------|------|
| NFR-PERF-001 | ライセンス照合処理時間 | 500 件の SW インベントリを **5 秒以内**に照合完了 | Celery タスク実行時間を Prometheus メトリクスで計測 | SAM-002 に対応。日次バッチ処理（03:00 実行） |
| NFR-PERF-002 | ダッシュボード表示速度 | 初期表示 **3 秒以内**（LCP） | Lighthouse Performance スコアで計測 | Grafana パネル埋め込み含む |
| NFR-PERF-003 | API レスポンスタイム | 95 パーセンタイルで **500ms 以内** | Prometheus histogram（http_request_duration_seconds） | SAM/調達/メトリクス API 全般 |
| NFR-PERF-004 | PWA オフライン表示速度 | キャッシュからの表示 **1 秒以内** | Service Worker レスポンス時間で計測 | Network First 戦略のフォールバック |
| NFR-PERF-005 | Prometheus スクレイプ | スクレイプ間隔 **60 秒**、ターゲット応答 **10 秒以内** | Prometheus targets ステータスで監視 | 500 台規模のエンドポイント対応 |
| NFR-PERF-006 | レポート生成時間 | J-SOX レポート生成 **30 秒以内** | タスク実行時間で計測 | PDF/CSV 出力 |

### 2.2 スループット

| 要件ID | 要件名 | 基準値 | 備考 |
|--------|--------|--------|------|
| NFR-PERF-010 | API 同時接続数 | **100 同時接続**に対応 | FastAPI + Uvicorn ワーカー設定 |
| NFR-PERF-011 | メトリクス受信スループット | **500 エンドポイント/分**のメトリクスを処理 | Prometheus リモートライト対応 |
| NFR-PERF-012 | Background Sync 処理 | オフライン蓄積 **100 件/回**の同期を処理 | PWA-003 対応 |

## 3. 可用性要件

| 要件ID | 要件名 | 基準値 | 詳細 | 対策 |
|--------|--------|--------|------|------|
| NFR-AVL-001 | サービス稼働率 | **99.5%以上**（月間ダウンタイム 3.65 時間以内） | AEGIS-SIGHT 本体と同一基準。計画メンテナンス時間は除外。 | Docker コンテナのヘルスチェック + 自動再起動、PostgreSQL のストリーミングレプリケーション |
| NFR-AVL-002 | Prometheus/Grafana 可用性 | **99.0%以上** | 監視基盤自体の可用性。短時間のデータ欠損は許容。 | Prometheus TSDB のローカル保持（15 日）、Grafana の永続ボリューム |
| NFR-AVL-003 | PWA オフライン可用性 | ネットワーク断絶時も**基本機能を継続提供** | 資産情報の参照・ステータス更新操作が可能 | Service Worker キャッシュ + IndexedDB によるオフラインストレージ |
| NFR-AVL-004 | 障害復旧時間（RTO） | **4 時間以内** | 障害発生から復旧完了まで | Docker Compose による迅速なサービス再構築、DB バックアップからの復元手順整備 |
| NFR-AVL-005 | 復旧時点目標（RPO） | **1 時間以内** | 障害時に許容されるデータ損失量 | PostgreSQL の WAL アーカイブ（1 時間間隔）、トランザクションログの定期バックアップ |
| NFR-AVL-006 | バックアップ | 日次フルバックアップ + 1 時間ごとの WAL アーカイブ | バックアップ保持期間：30 日 | pg_basebackup + WAL-G による自動バックアップ |

## 4. セキュリティ要件

### 4.1 アクセス制御

| 要件ID | 要件名 | 基準値 | 詳細 | 準拠規格 |
|--------|--------|--------|------|----------|
| NFR-SEC-001 | 費用情報アクセス制限 | 管理者ロール（admin）**のみ**アクセス可 | 調達費用・保守費用・単価情報は RBAC で制御。一般ユーザには金額フィールドを非表示。 | ISO27001:2022 A.5.15 アクセス制御 |
| NFR-SEC-002 | ロールベースアクセス制御（RBAC） | 4 ロール：admin / manager / operator / viewer | ロール定義：admin（全権限）、manager（承認権限）、operator（申請・登録権限）、viewer（閲覧のみ） | ISO27001:2022 A.5.15, A.8.3 |
| NFR-SEC-003 | API 認証 | **JWT + OAuth 2.0** | アクセストークン有効期限 15 分、リフレッシュトークン 7 日 | ISO27001:2022 A.8.5 |
| NFR-SEC-004 | Prometheus エンドポイント保護 | **社内ネットワーク限定**公開 | Nginx リバースプロキシで外部アクセスをブロック。基本認証を追加。 | NIST CSF 2.0 PROTECT PR.AC |

### 4.2 データ保護

| 要件ID | 要件名 | 基準値 | 詳細 | 準拠規格 |
|--------|--------|--------|------|----------|
| NFR-SEC-010 | ライセンスキー暗号化 | **AES-256-GCM** | Azure Key Vault でキー管理。DB 保存時に暗号化。 | ISO27001:2022 A.8.24 暗号の使用 |
| NFR-SEC-011 | 通信暗号化 | **TLS 1.2 以上** | クライアント - サーバ間の全通信を暗号化 | ISO27001:2022 A.8.24 |
| NFR-SEC-012 | PWA キャッシュデータ制限 | **個人識別情報を含まない**設計 | Service Worker キャッシュには資産のサマリ情報のみ保存。ライセンスキー・費用情報はキャッシュ対象外。 | ISO27001:2022 A.5.34 プライバシー |
| NFR-SEC-013 | 廃棄証明書保護 | Azure Blob Storage に暗号化保存 | サーバサイド暗号化（SSE）、アクセスは管理者ロールのみ | J-SOX IT 全般統制 |
| NFR-SEC-014 | 監査ログ | 全操作の監査ログを記録 | 操作者・日時・対象リソース・操作内容を記録。改ざん防止のためログに署名。 | ISO27001:2022 A.8.15 ログ取得 |

### 4.3 脆弱性管理

| 要件ID | 要件名 | 基準値 | 詳細 |
|--------|--------|--------|------|
| NFR-SEC-020 | 依存パッケージ脆弱性スキャン | CI パイプラインで**毎ビルド実行** | Trivy / Safety / npm audit による自動スキャン |
| NFR-SEC-021 | コンテナイメージスキャン | デプロイ前に**毎回実行** | Trivy によるイメージスキャン、Critical/High は修正必須 |
| NFR-SEC-022 | 静的解析（SAST） | **毎 PR** | Bandit（Python）/ ESLint Security Plugin（TypeScript） |

## 5. スケーラビリティ要件

| 要件ID | 要件名 | 基準値 | 詳細 | 拡張方針 |
|--------|--------|--------|------|----------|
| NFR-SCL-001 | 管理端末数 | **500 台以上** | Windows 11/10 クライアント PC | エージェントの file_sd_configs による動的ターゲット管理。1,000 台まではスケールアップ、以降は Prometheus フェデレーション |
| NFR-SCL-002 | 管理ライセンス数 | **1,000 件以上** | ソフトウェアライセンス登録数 | PostgreSQL のインデックス最適化。5,000 件まで単一インスタンスで対応 |
| NFR-SCL-003 | PWA キャッシュサイズ | **最大 50MB** | 直近 30 日分の資産情報 | キャッシュ容量監視、古いデータの自動パージ（LRU） |
| NFR-SCL-004 | Prometheus データ保持 | ローカル **15 日**保持 | Prometheus TSDB のリテンション設定 | 長期保存が必要な場合は Thanos / Cortex を導入 |
| NFR-SCL-005 | 調達申請数 | **年間 500 件以上** | 年間の調達申請処理量 | ステータスインデックス、requester_id インデックスで検索高速化 |
| NFR-SCL-006 | 同時利用ユーザ数 | **50 ユーザ** | ダッシュボード・管理画面の同時利用 | Next.js SSR + CDN キャッシュ、Uvicorn ワーカー数調整 |

## 6. 保守性要件

### 6.1 コード品質

| 要件ID | 要件名 | 基準値 | 詳細 |
|--------|--------|--------|------|
| NFR-MNT-001 | テストカバレッジ | **80%以上**（統合後全体） | バックエンド pytest-cov、フロントエンド Vitest coverage で計測。各モジュール最低 70%。 |
| NFR-MNT-002 | コードスタイル | **Linter/Formatter 準拠** | Python: Ruff（Black + isort 互換）、TypeScript: ESLint + Prettier。CI で自動チェック。 |
| NFR-MNT-003 | 型安全性 | **静的型チェック合格** | Python: mypy strict mode、TypeScript: strict mode。CI でビルドエラーとして検出。 |
| NFR-MNT-004 | ドキュメント | **API ドキュメント自動生成** | FastAPI の OpenAPI 自動生成 + Swagger UI。変更時に自動更新。 |

### 6.2 運用保守

| 要件ID | 要件名 | 基準値 | 詳細 |
|--------|--------|--------|------|
| NFR-MNT-010 | ログ保存期間 | **7 年**（調達・廃棄記録） | J-SOX IT 全般統制対応。操作ログ・監査証跡は改ざん防止措置付き。 |
| NFR-MNT-011 | アプリケーションログ | **90 日**保持 | エラーログ・アクセスログ。ELK / Loki で集約。 |
| NFR-MNT-012 | ログレベル | **構造化ログ（JSON）** | タイムスタンプ・リクエストID・ユーザID・操作内容を含む。ログレベル：DEBUG/INFO/WARNING/ERROR/CRITICAL。 |
| NFR-MNT-013 | デプロイ方式 | **Docker Compose / CI/CD** | GitHub Actions による自動デプロイ。ブルーグリーンデプロイまたはローリングアップデート。ロールバック 5 分以内。 |
| NFR-MNT-014 | 設定管理 | **環境変数 + シークレット管理** | 環境固有設定は .env ファイル、シークレットは Azure Key Vault。ハードコーディング禁止。 |
| NFR-MNT-015 | DB マイグレーション | **Alembic による自動マイグレーション** | マイグレーションスクリプトのバージョン管理。ロールバック可能な設計。 |

### 6.3 監視・アラート

| 要件ID | 要件名 | 基準値 | 詳細 |
|--------|--------|--------|------|
| NFR-MNT-020 | ヘルスチェック | 全コンテナに**/health エンドポイント** | HTTP 200 応答で正常判定。Docker ヘルスチェック + Prometheus blackbox_exporter。 |
| NFR-MNT-021 | アラート通知チャネル | **メール + Slack** | Alertmanager 経由。severity: critical はメール + Slack、warning は Slack のみ。 |
| NFR-MNT-022 | SLA ダッシュボード | Grafana で**稼働率・性能 SLA を可視化** | 月次稼働率、API 応答時間 P95、照合処理時間のトレンド表示。 |
