#### 移植対象の選定基準

| 基準 | 移植する | 移植しない |
|------|----------|------------|
| AEGIS-SIGHTの要件との合致 | Prometheus/Grafana、PWA、調達、SAM | - |
| ITSM-Systemとの重複 | - | CMDB、インシデント管理、SLA管理、変更管理 |
| 技術的移植可能性 | Node.js→Python変換可能 | 複雑すぎる依存関係 |
| ビジネス価値 | エンドポイント管理に直結 | 間接的な管理機能 |

#### API設計原則

- RESTful設計（OpenAPI 3.1準拠）
- バージョニング: `/api/v1/`プレフィックス
- 認証: OAuth 2.0 + JWT（M365 Azure AD連携）
- レート制限: 100 req/min/user
- ページネーション: カーソルベース
- エラーレスポンス: RFC 7807 Problem Details形式

## 5. リスクと対策

| リスク | 対策 |
|--------|------|
| IAMSコードのドキュメント不足 | コード自体をSingle Source of Truthとし、静的解析ツールで補完 |
| 移植対象の判断ミス | 選定基準を明確化し、ステークホルダーレビューで合意 |
| 設計工数の超過 | タイムボックスを厳守、完璧よりも十分な設計を優先 |
| チームのIAMS理解不足 | ペアプログラミングでの知識共有、設計セッションの実施 |
