#### 調達ワークフロー状態遷移

```
[申請作成] → [申請中] → [承認待ち] → [承認済み] → [発注済み] → [納品待ち] → [検収完了]
                ↓           ↓                                        ↓
            [差し戻し]   [却下]                                   [返品]
```

#### 調達管理 API エンドポイント設計

```
GET    /api/v1/procurement/requests              # 調達申請一覧
POST   /api/v1/procurement/requests              # 調達申請作成
GET    /api/v1/procurement/requests/{id}         # 申請詳細
PUT    /api/v1/procurement/requests/{id}         # 申請更新
POST   /api/v1/procurement/requests/{id}/submit  # 申請提出
POST   /api/v1/procurement/requests/{id}/approve # 承認
POST   /api/v1/procurement/requests/{id}/reject  # 却下
POST   /api/v1/procurement/requests/{id}/return  # 差し戻し
GET    /api/v1/procurement/orders                # 発注一覧
POST   /api/v1/procurement/orders                # 発注作成
GET    /api/v1/procurement/orders/{id}           # 発注詳細
POST   /api/v1/procurement/deliveries            # 納品登録
POST   /api/v1/procurement/deliveries/{id}/accept # 検収完了
```

## 5. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| IAMS SAMコードの移植困難 | 高 | Phase1の分析結果に基づき、移植 vs 新規実装を判断 |
| M365 API rate limit | 中 | バッチ処理、キャッシュ、リトライ機構の実装 |
| 承認ワークフローの複雑化 | 中 | 状態遷移図の事前合意、段階的実装 |
| フロント・バック連携の不整合 | 中 | OpenAPI仕様を契約とし、モック先行開発 |
