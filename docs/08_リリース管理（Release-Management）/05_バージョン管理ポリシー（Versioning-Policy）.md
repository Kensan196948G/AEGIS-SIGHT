## 1. Semantic Versioning（SemVer）

AEGIS-SIGHTは**Semantic Versioning 2.0.0**（セマンティックバージョニング）を採用する。

### 1.1 バージョン形式

```
MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]

例:
  1.0.0           正式リリース
  1.1.0           機能追加
  1.1.1           バグ修正
  2.0.0           破壊的変更
  1.0.0-alpha.1   アルファ版
  1.0.0-beta.1    ベータ版
  1.0.0-rc.1      リリース候補
```

### 1.2 バージョン番号のルール

| 要素 | インクリメント条件 | 例 |
|------|-----------------|-----|
| MAJOR | 後方互換性のない変更（API変更、DB破壊的変更） | 1.0.0 → 2.0.0 |
| MINOR | 後方互換性のある機能追加 | 1.0.0 → 1.1.0 |
| PATCH | 後方互換性のあるバグ修正 | 1.0.0 → 1.0.1 |

### 1.3 プレリリース識別子

| 識別子 | 用途 | 例 |
|--------|------|-----|
| alpha | 開発初期、機能不完全 | 0.1.0-alpha.1 |
| beta | 機能完成、品質検証中 | 0.3.0-beta.1 |
| rc | リリース候補、最終検証 | 1.0.0-rc.1 |

## 2. ブランチ戦略

### 2.1 ブランチモデル

```
main ──────────────────────────────────────────────
  │                    ↑ merge          ↑ merge
  │              release/v1.0.0    release/v1.1.0
  │                    ↑                ↑
develop ──────────────────────────────────────────
  │     ↑        ↑        ↑       ↑
  │  feature/  feature/  feature/ bugfix/
  │  auth      devices   iams     fix-login
  │
  └── hotfix/critical-fix ──→ main + develop
```

### 2.2 ブランチ命名規約

| ブランチ種別 | 命名パターン | 用途 | ベース |
|-------------|-------------|------|--------|
| main | `main` | 本番リリース済みコード | - |
| develop | `develop` | 開発統合ブランチ | main |
| feature | `feature/<機能名>` | 新機能開発 | develop |
| bugfix | `bugfix/<修正内容>` | バグ修正 | develop |
| release | `release/v{X.Y.Z}` | リリース準備 | develop |
| hotfix | `hotfix/<修正内容>` | 本番緊急修正 | main |

### 2.3 ブランチとバージョンの関係

| ブランチ | バージョン例 | 説明 |
|---------|------------|------|
| main | v1.0.0, v1.1.0 | リリースタグ付き |
| develop | 1.2.0-dev | 次バージョン開発中 |
| release/v1.1.0 | 1.1.0-rc.1 | リリース候補 |
| feature/* | バージョン不要 | 機能開発中 |
| hotfix/* | 1.0.1 | パッチバージョン |

## 3. タグ戦略

### 3.1 タグ命名規約

| タグ形式 | 用途 | 例 |
|---------|------|-----|
| `v{MAJOR}.{MINOR}.{PATCH}` | 正式リリース | v1.0.0 |
| `v{MAJOR}.{MINOR}.{PATCH}-alpha.{N}` | アルファリリース | v0.1.0-alpha.1 |
| `v{MAJOR}.{MINOR}.{PATCH}-beta.{N}` | ベータリリース | v0.3.0-beta.1 |
| `v{MAJOR}.{MINOR}.{PATCH}-rc.{N}` | リリース候補 | v1.0.0-rc.1 |

### 3.2 タグ作成手順

```bash
# 正式リリースタグ
git tag -a v1.0.0 -m "Release v1.0.0: 正式リリース"
git push origin v1.0.0

# プレリリースタグ
git tag -a v1.0.0-rc.1 -m "Release Candidate v1.0.0-rc.1"
git push origin v1.0.0-rc.1
```

## 4. バージョン管理対象

### 4.1 バージョンを管理するファイル

| ファイル | 用途 | 更新タイミング |
|---------|------|-------------|
| `backend/app/version.py` | バックエンドバージョン | リリースブランチ作成時 |
| `frontend/package.json` | フロントエンドバージョン | リリースブランチ作成時 |
| `docker-compose.yml` | イメージタグ | リリースブランチ作成時 |
| `CHANGELOG.md` | 変更履歴 | 継続的（リリース時確定） |

### 4.2 バージョン記載例

```python
# backend/app/version.py

```

```json
// frontend/package.json
{
  "name": "aegis-sight-frontend",
  "version": "1.0.0"
}
```

## 5. APIバージョニング

### 5.1 APIバージョン方針

AEGIS-SIGHTのAPIはURLパスにバージョンを含める。

```
/api/v1/users
/api/v1/devices
/api/v2/users    ← 破壊的変更時に新バージョン
```

### 5.2 APIバージョン互換性ルール

| 変更種別 | APIバージョン | アプリバージョン |
|---------|-------------|----------------|
| フィールド追加（レスポンス） | 同バージョン | MINOR |
| オプションパラメータ追加 | 同バージョン | MINOR |
| 必須パラメータ追加 | 新バージョン | MAJOR |
| フィールド削除 | 新バージョン | MAJOR |
| レスポンス形式変更 | 新バージョン | MAJOR |
| エンドポイント追加 | 同バージョン | MINOR |
| エンドポイント削除 | 新バージョン | MAJOR |

### 5.3 APIバージョン廃止ポリシー

| 段階 | 期間 | アクション |
|------|------|----------|
| 非推奨通知 | 廃止3ヶ月前 | レスポンスヘッダーに `Deprecation` 追加 |
| 移行期間 | 3ヶ月 | v1 と v2 を並行稼働 |
| 廃止 | 移行期間終了後 | v1 を削除、410 Gone を返却 |

## 6. データベーススキーマバージョニング

### 6.1 マイグレーション管理

| ツール | Alembic |
|--------|---------|
| マイグレーションディレクトリ | `backend/alembic/versions/` |
| 命名規約 | `{revision}_{description}.py` |

### 6.2 マイグレーションルール

| ルール | 説明 |
|--------|------|
| 前方互換性優先 | 可能な限り後方互換なスキーマ変更 |
| ダウングレード必須 | 全マイグレーションにdowngradeを実装 |
| データマイグレーション分離 | スキーマ変更とデータ変更を別マイグレーションに |
| テスト必須 | マイグレーションのup/downをテスト |

## 7. 依存関係バージョン管理

### 7.1 バックエンド（Python）

```toml
# pyproject.toml - バージョン固定方針
[project]
dependencies = [
    "fastapi>=0.110,<1.0",      # MINOR範囲で許容
    "sqlalchemy>=2.0,<3.0",     # MAJOR固定
    "pydantic>=2.0,<3.0",       # MAJOR固定
    "alembic>=1.13,<2.0",       # MAJOR固定
]
```

### 7.2 フロントエンド（Node.js）

```json
// package.json - バージョン固定方針
{
  "dependencies": {
    "next": "^14.0.0",
    "react": "^18.0.0",
    "typescript": "^5.0.0"
  }
}
```

### 7.3 依存関係更新ポリシー

| 更新種別 | 頻度 | 方法 |
|---------|------|------|
| セキュリティパッチ | 即時 | Dependabot / 手動 |
| PATCH更新 | 週次 | Dependabot |
| MINOR更新 | 月次 | 手動確認後適用 |
| MAJOR更新 | 四半期 | 影響調査後計画的に適用 |

## 8. バージョン判定フローチャート

```
変更内容を確認
  │
  ├── 既存APIの互換性を壊す？
  │     ├── Yes → MAJOR バージョンアップ
  │     └── No ──┐
  │              │
  │   新機能の追加？
  │     ├── Yes → MINOR バージョンアップ
  │     └── No ──┐
  │              │
  │   バグ修正のみ？
  │     ├── Yes → PATCH バージョンアップ
  │     └── No → 変更なし（ドキュメントのみ等）
```
