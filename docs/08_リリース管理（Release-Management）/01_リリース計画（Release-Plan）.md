## 3. リリースプロセス

### 3.1 リリースフロー

```
1. Feature Freeze（機能凍結）
   └── 新機能マージ停止、バグ修正のみ

2. Release Branch作成
   └── release/v{X.Y.Z} ブランチ作成

3. QA検証
   ├── 統合テスト実行
   ├── E2Eテスト実行
   ├── セキュリティスキャン
   └── パフォーマンステスト

4. Release Candidate
   ├── RCタグ付与（v{X.Y.Z}-rc.{N}）
   ├── ステージング環境デプロイ
   └── 受入テスト

5. Release
   ├── リリースタグ付与（v{X.Y.Z}）
   ├── リリースノート作成
   ├── 本番デプロイ
   └── 監視強化（リリース後24時間）

6. Post-Release
   ├── ホットフィックス対応（必要時）
   ├── リリースレトロスペクティブ
   └── 次バージョン計画開始
```

### 3.2 リリースサイクル

| フェーズ | 期間 | 活動 |
|---------|------|------|
| 開発 | 2-4週間 | 機能開発、単体テスト |
| Feature Freeze | 3日 | マージ停止、バグ修正 |
| QA | 3-5日 | 統合テスト、E2E、セキュリティ |
| RC | 2-3日 | ステージング検証、受入テスト |
| リリース | 1日 | デプロイ、監視 |
| 安定化 | 3日 | ホットフィックス対応 |

## 4. リリースノート形式

### 4.1 テンプレート

```markdown
# AEGIS-SIGHT v{X.Y.Z} リリースノート

**リリース日**: YYYY-MM-DD
**リリースマネージャー**: <担当者名>

## ハイライト

<このリリースの主要な変更点を1-3文で説明>

## 新機能

- **<機能名>**: <説明> (#<Issue番号>)
- **<機能名>**: <説明> (#<Issue番号>)

## 改善

- <改善内容> (#<Issue番号>)
- <改善内容> (#<Issue番号>)

## バグ修正

- <修正内容> (#<Issue番号>)
- <修正内容> (#<Issue番号>)

## 破壊的変更

- <変更内容と移行方法>

## 既知の問題

- <問題と回避策>

## アップグレード手順

1. <手順1>
2. <手順2>

## 変更統計

- コミット数: XX
- 変更ファイル数: XX
- コントリビューター数: XX
```

### 4.2 リリースノート記載基準

| 種類 | 記載基準 | 例 |
|------|---------|-----|
| 新機能 | ユーザーに見える新機能 | デバイス一括インポート機能を追加 |
| 改善 | 既存機能の改善 | デバイス一覧の表示速度を50%改善 |
| バグ修正 | ユーザー影響のあるバグ | ログイン時にセッションが保持されない問題を修正 |
| 破壊的変更 | API変更、DB変更等 | ユーザーAPIのレスポンス形式変更 |
| セキュリティ | セキュリティ修正 | XSS脆弱性の修正（CVE-XXXX-XXXX） |

## 5. リリース承認フロー

| 段階 | 承認者 | 確認事項 |
|------|--------|---------|
| Feature Freeze | テックリード | 機能完成度、テストカバレッジ |
| QA完了 | QAリード | テスト結果、不具合ゼロ |
| RC承認 | プロジェクトマネージャー | 受入テスト結果、リスク評価 |
| リリース承認 | CTO / プロダクトオーナー | 最終判断 |
| デプロイ実行 | インフラ担当 | デプロイ手順確認 |

---

## 6. ホットフィックスポリシー

| 重要度 | 対応時間 | プロセス |
|--------|---------|---------|
| Critical（サービス停止） | 2時間以内 | 即座にhotfix/*ブランチ作成→修正→テスト→デプロイ |
| High（主要機能障害） | 24時間以内 | hotfix/*ブランチ→通常テスト→デプロイ |
| Medium | 次スプリント | 通常開発フローに組み込み |
| Low | バックログ | 優先度に応じて対応 |
