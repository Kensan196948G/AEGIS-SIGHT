## 1. 目的

本コミュニケーション計画は、プロジェクトに関わる全ステークホルダーに対して、適切な情報を適切なタイミングで提供し、プロジェクトの円滑な推進と意思決定の迅速化を実現するためのコミュニケーション体制・手段・ルールを定義するものである。

---

## 2. コミュニケーション方針

### 2.1 基本原則

| 原則 | 説明 |
|------|------|
| **透明性** | プロジェクトの進捗・課題・リスクをタイムリーかつ正確に共有する |
| **適時性** | 必要な情報を必要なタイミングで提供する。遅延した情報共有はリスクとなる |
| **適切性** | ステークホルダーの役割・関心度に応じた情報の粒度・形式で提供する |
| **双方向性** | 一方的な報告ではなく、フィードバックを積極的に収集・反映する |
| **記録性** | 重要な意思決定・合意事項は文書化し、トレーサビリティを確保する |

### 2.2 コミュニケーション手段

| 手段 | 用途 | ツール |
|------|------|--------|
| 対面会議 | 重要な意思決定、キックオフ、レビュー | 会議室 / Microsoft Teams |
| オンライン会議 | 定例会議、リモート参加者がいる場合 | Microsoft Teams |
| チャット | 日常的な情報共有、質疑応答、速報 | Slack（プロジェクト専用チャンネル） |
| メール | 公式通知、承認依頼、外部ステークホルダー連絡 | Microsoft Outlook |
| 文書共有 | 設計書、議事録、レポートの共有・管理 | GitHub（リポジトリ内 docs/）、SharePoint |
| プロジェクト管理 | タスク管理、進捗追跡、課題管理 | GitHub Projects / Issues |
| CI/CD 通知 | ビルド・テスト結果の自動通知 | GitHub Actions → Slack 連携 |

## 3. 報告体制

### 3.1 報告ライン

```
経営企画部長（スポンサー）
    ▲ 月次経営報告
    │
IT部門長（承認者）
    ▲ 週次ステータスレポート
    │
PM（プロジェクトマネージャー）
    ▲ 日次スタンドアップ報告
    │
    ├── TL（テクニカルリード）
    │     ├── BE開発者
    │     ├── FE開発者
    │     └── インフラ担当
    │
    └── QAリード
          └── テストメンバー
```

### 3.2 報告種別と責任

| 報告種別 | 作成者 | 承認者 | 配布先 | 頻度 |
|---------|--------|--------|--------|------|
| 日次進捗報告 | 各メンバー | TL | PM、チーム全員 | 毎日 |
| 週次ステータスレポート | PM | IT部門長 | スポンサー、IT部門長、チーム | 毎週金曜 |
| 月次経営報告 | PM | IT部門長 | 経営企画部長 | 毎月末 |
| フェーズ完了報告 | PM | IT部門長 | 全ステークホルダー | 各フェーズ終了時 |
| リスク報告 | PM | IT部門長 | 関係者 | 発生時即時 |

## 4. 定例会議

### 4.1 会議体系

| 会議名 | 目的 | 参加者 | 頻度 | 時間 | 形式 |
|--------|------|--------|------|------|------|
| デイリースタンドアップ | 日次進捗確認、ブロッカー共有 | PM, TL, QA, 開発メンバー | 毎日 | 15分 | 対面/オンライン |
| 週次プロジェクト会議 | 進捗・課題・リスクの総合レビュー | PM, TL, QA, IT部門長 | 毎週水曜 | 60分 | 対面/オンライン |
| スプリントレビュー | 成果物デモ、フィードバック収集 | PM, TL, QA, 開発メンバー, IT部門長 | 隔週金曜 | 90分 | 対面 |
| スプリントレトロスペクティブ | プロセス改善、振り返り | PM, TL, QA, 開発メンバー | 隔週金曜（レビュー後） | 60分 | 対面 |
| 月次経営報告会 | スポンサーへの進捗報告、経営判断事項 | PM, IT部門長, 経営企画部長 | 月末 | 30分 | 対面 |
| フェーズゲートレビュー | フェーズ完了判定、次フェーズ開始承認 | PM, IT部門長, TL, QA, 関係者 | 各フェーズ終了時 | 120分 | 対面 |
| セキュリティレビュー | セキュリティ設計・テスト結果確認 | TL, QA, セキュリティ担当 | Phase 1, Phase 4 | 90分 | 対面 |
| 運用引き継ぎ会議 | 運用チームへの引き継ぎ | PM, TL, IT運用チーム | Phase 3以降隔週 | 60分 | 対面 |

### 4.2 会議運営ルール

| ルール | 内容 |
|--------|------|
| アジェンダ事前配布 | 会議の24時間前までにアジェンダを配布する |
| タイムキーパー | 各会議にタイムキーパーを設置し、時間厳守とする |
| 議事録作成 | 会議終了後24時間以内に議事録を作成・配布する |
| アクションアイテム | 決定事項・アクションアイテムは担当者・期限を明記する |
| 欠席時の代理 | 欠席の場合は事前に代理出席者を指名し、事前資料で意見を共有する |
| 録画 | オンライン会議は録画し、欠席者が確認可能にする |

### 4.3 デイリースタンドアップの構成

各メンバーが以下の3点を報告する（1人2分以内）。

| 項目 | 内容 |
|------|------|
| 昨日やったこと | 完了タスク、進捗のあったタスク |
| 今日やること | 本日着手・継続するタスク |
| ブロッカー | 作業を妨げている課題、支援が必要な事項 |

## 6. エスカレーション手順

### 6.1 エスカレーションレベル

| レベル | トリガー条件 | エスカレーション先 | 応答時間 |
|:------:|-------------|-------------------|---------|
| L1 | 日常的な技術的課題、軽微な遅延 | TL / QAリード | 4時間以内 |
| L2 | 解決困難な技術的課題、1週間以内の遅延 | PM | 1営業日以内 |
| L3 | スケジュール遅延（1週間超）、品質基準未達 | IT部門長 | 1営業日以内 |
| L4 | プロジェクト継続に関わる重大問題、セキュリティインシデント | 経営企画部長 | 即時（2時間以内） |

### 6.2 エスカレーションフロー

```
問題発生
    │
    ▼
L1: 担当者 → TL/QAリード（4時間）
    │ 解決？→ Yes → クローズ
    │ No
    ▼
L2: TL/QAリード → PM（1営業日）
    │ 解決？→ Yes → クローズ
    │ No
    ▼
L3: PM → IT部門長（1営業日）
    │ 解決？→ Yes → クローズ
    │ No
    ▼
L4: IT部門長 → 経営企画部長（即時）
    │ → 緊急対策会議召集
    ▼
    クローズ
```

### 6.3 エスカレーション判断基準

| カテゴリ | L1 → L2 基準 | L2 → L3 基準 | L3 → L4 基準 |
|---------|-------------|-------------|-------------|
| スケジュール | タスク遅延 2日以上 | フェーズ遅延 1週間以上 | プロジェクト遅延 2週間以上 |
| 品質 | 単体テスト失敗 5件以上 | カバレッジ 70%未満 | セキュリティ脆弱性 Critical 検出 |
| リソース | メンバー1名の稼働率低下 | チーム全体の稼働率80%未満 | キーパーソン離脱 |
| セキュリティ | 軽微な脆弱性検出 | High 脆弱性検出 | Critical 脆弱性 / インシデント |
| コンプライアンス | 軽微な不適合 | ISO/J-SOX要件の不適合 | 監査指摘（重大） |
