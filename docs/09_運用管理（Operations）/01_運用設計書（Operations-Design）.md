## 1. 運用体制

### 1.1 組織構成

```
┌─────────────────────────────────┐
│         CTO / 情報システム部長     │
│         （最終意思決定）           │
└──────────┬──────────────────────┘
           │
    ┌──────┴──────┐
    │             │
┌───┴───┐   ┌───┴────┐
│開発チーム│   │運用チーム │
│        │   │         │
│・機能開発│   │・定常運用 │
│・バグ修正│   │・監視対応 │
│・テスト  │   │・障害対応 │
└────────┘   │・バックアップ│
             └──────────┘
```

### 1.2 役割と責任

| 役割 | 担当者数 | 主な責任 |
|------|---------|---------|
| 運用マネージャー | 1名 | 運用全体管理、エスカレーション判断 |
| インフラ担当 | 1-2名 | サーバー管理、デプロイ、監視 |
| アプリ運用担当 | 1-2名 | アプリケーション監視、ログ分析 |
| DB管理者 | 1名 | データベース管理、バックアップ、チューニング |
| セキュリティ担当 | 1名（兼任可） | セキュリティ監視、脆弱性対応 |

### 1.3 対応時間

| 区分 | 対応時間 | 対応内容 |
|------|---------|---------|
| 通常営業時間 | 平日 9:00-18:00 | 全運用業務 |
| 時間外（準備対応） | 平日 18:00-22:00 | 重大障害のみ対応 |
| 休日・深夜 | 土日祝、22:00-9:00 | 自動監視 + 重大障害のみ |

## 2. 定常運用

### 2.1 日次運用

| # | 作業 | 時間 | 担当 | 手順 |
|---|------|------|------|------|
| 1 | 監視ダッシュボード確認 | 9:00 | インフラ担当 | Grafanaダッシュボードを確認 |
| 2 | 前日のアラート確認 | 9:15 | アプリ運用担当 | アラート一覧をレビュー |
| 3 | ログ異常チェック | 9:30 | アプリ運用担当 | エラーログ件数を確認 |
| 4 | バックアップ成否確認 | 10:00 | DB管理者 | バックアップログを確認 |
| 5 | ディスク使用量確認 | 10:00 | インフラ担当 | 閾値（80%）未満を確認 |
| 6 | 日次運用レポート作成 | 17:00 | 運用マネージャー | 運用状況を記録 |

### 2.2 週次運用

| # | 作業 | 曜日 | 担当 | 手順 |
|---|------|------|------|------|
| 1 | セキュリティスキャン | 月曜 | セキュリティ担当 | 脆弱性スキャン実行・結果確認 |
| 2 | パフォーマンスレビュー | 水曜 | インフラ担当 | レスポンス時間・リソース使用率の傾向分析 |
| 3 | DB統計情報更新 | 木曜 | DB管理者 | ANALYZE実行 |
| 4 | ログローテーション確認 | 金曜 | インフラ担当 | ログファイルサイズ確認 |
| 5 | 週次運用レポート | 金曜 | 運用マネージャー | 週次サマリー作成 |

### 2.3 月次運用

| # | 作業 | 時期 | 担当 | 手順 |
|---|------|------|------|------|
| 1 | OS・ミドルウェアパッチ適用 | 第2水曜 | インフラ担当 | ステージング検証後本番適用 |
| 2 | SSL証明書有効期限確認 | 月初 | インフラ担当 | 有効期限 > 60日を確認 |
| 3 | バックアップ復元テスト | 月末 | DB管理者 | ステージングで復元テスト |
| 4 | 容量計画レビュー | 月末 | インフラ担当 | ストレージ・CPU・メモリの傾向分析 |
| 5 | アカウント棚卸 | 月末 | セキュリティ担当 | 不要アカウントの無効化 |
| 6 | 月次運用レポート | 月末 | 運用マネージャー | 月次サマリー・SLA実績報告 |

## 3. 保守作業

### 3.1 定期保守

| 保守作業 | 頻度 | 想定時間 | ダウンタイム |
|---------|------|---------|------------|
| OSパッチ適用 | 月次 | 30分 | なし（ローリング） |
| Dockerイメージ更新 | 月次 | 15分 | なし |
| DB VACUUM/ANALYZE | 週次 | 10分 | なし |
| ログ圧縮・アーカイブ | 週次 | 5分 | なし |
| 不要Dockerイメージ削除 | 月次 | 5分 | なし |
| SSL証明書更新 | 年次 | 10分 | なし |
| PostgreSQLメジャーアップデート | 年次 | 2時間 | あり（計画保守窓） |

### 3.2 計画保守窓

| 項目 | 内容 |
|------|------|
| 定期保守窓 | 毎月第2日曜 2:00-6:00 |
| 緊急保守窓 | 随時（CTO承認要） |
| 通知タイミング | 定期: 5営業日前、緊急: 可能な限り事前 |
| 通知方法 | メール + チャット |

### 3.3 保守作業手順テンプレート

```
保守作業名: _______________
実施日時: YYYY-MM-DD HH:MM - HH:MM
担当者: _______________
影響範囲: _______________
ダウンタイム: あり / なし（想定: XX分）

事前作業:
1. □ バックアップ取得
2. □ 関係者通知
3. □ ロールバック手順確認

実施手順:
1. □ _______________
2. □ _______________
3. □ _______________

事後確認:
1. □ ヘルスチェック
2. □ ログ確認
3. □ 動作確認
4. □ 完了通知
```

## 6. 変更管理

### 6.1 変更管理フロー

```
変更要求 → 影響評価 → 承認 → 実施計画 → 実施 → 検証 → クローズ
```

### 6.2 変更カテゴリ

| カテゴリ | 承認者 | リードタイム | 例 |
|---------|--------|------------|-----|
| 標準変更 | 運用マネージャー | 1営業日 | 設定変更、パッチ適用 |
| 通常変更 | テックリード | 3営業日 | 機能追加、構成変更 |
| 緊急変更 | CTO | 即時 | セキュリティパッチ、障害対応 |
