## 1. 障害レベル定義

| レベル | 名称 | 定義 | 影響範囲 | 対応目標時間 |
|--------|------|------|---------|------------|
| P1 | Critical | サービス全停止、データ損失の恐れ | 全ユーザー | 検知後15分以内に対応開始、1時間以内に復旧 |
| P2 | Major | 主要機能が利用不可 | 多数ユーザー | 検知後30分以内に対応開始、4時間以内に復旧 |
| P3 | Minor | 一部機能の不具合、回避策あり | 一部ユーザー | 翌営業日中に対応 |
| P4 | Low | 軽微な不具合、運用に影響なし | 限定的 | 次スプリントで対応 |

### 1.1 障害レベル判定マトリクス

| 影響 ＼ 緊急度 | 即座に対応必要 | 数時間以内 | 翌営業日 |
|---------------|-------------|-----------|---------|
| 全ユーザー影響 | **P1** | P1 | P2 |
| 多数ユーザー影響 | P1 | **P2** | P3 |
| 一部ユーザー影響 | P2 | P3 | **P3** |
| 限定的影響 | P3 | P3 | **P4** |

### 1.2 障害例

| レベル | 例 |
|--------|-----|
| P1 | サーバー停止、DB接続不可、認証機能停止、データ破損 |
| P2 | デバイス管理画面エラー、レポート生成失敗、極端な応答遅延 |
| P3 | 特定ブラウザでのUI崩れ、CSVエクスポートのフォーマット不正 |
| P4 | 誤字脱字、UIの軽微なデザイン不具合 |

## 2. 障害対応フロー

### 2.1 全体フロー

```
検知 → トリアージ → エスカレーション → 調査 → 対応 → 復旧確認 → 事後対応
  │        │            │              │       │        │           │
  │     レベル判定    通知先決定     根本原因特定  修正適用  正常性確認   ポストモーテム
  │        │            │              │       │        │           │
  ▼        ▼            ▼              ▼       ▼        ▼           ▼
アラート   P1-P4     関係者通知     ログ分析   デプロイ  ヘルスチェック  RCA作成
or 報告   分類      チケット作成   メトリクス  or 設定    監視強化     再発防止
```

### 2.2 P1（Critical）対応フロー

```
1. [0分]   アラート受信 / 障害報告
2. [5分]   第一報を運用チャンネルに投稿
           - 障害概要
           - 影響範囲（推定）
           - 対応者
3. [10分]  テックリード・CTOにエスカレーション
4. [15分]  インシデントコマンダー決定
           - 調査担当アサイン
           - コミュニケーション担当アサイン
5. [15-60分] 調査・対応実施
           - ログ分析
           - メトリクス確認
           - 修正 or ロールバック判断
6. [復旧]  復旧確認
           - ヘルスチェック
           - 動作確認
           - ユーザー影響の確認
7. [復旧後] 事後処理
           - 全関係者に復旧通知
           - インシデントチケット更新
           - 24時間監視強化
8. [3営業日以内] ポストモーテム実施
```

### 2.3 P2（Major）対応フロー

```
1. [0分]    アラート受信 / 障害報告
2. [15分]   障害チケット作成、運用チャンネルに投稿
3. [30分]   テックリードにエスカレーション
4. [30-240分] 調査・対応実施
5. [復旧]   復旧確認・通知
6. [5営業日以内] ポストモーテム実施
```

## 4. 障害調査手順

### 4.1 初動調査チェックリスト

```bash
# 1. サービス状態確認
docker compose ps
curl -s http://localhost:8000/api/v1/health | jq .
curl -s http://localhost:8000/api/v1/health/ready | jq .

# 2. コンテナログ確認（直近のエラー）
docker compose logs --tail=100 backend 2>&1 | grep -i "error\|exception\|fatal"
docker compose logs --tail=100 frontend 2>&1 | grep -i "error"
docker compose logs --tail=50 db 2>&1 | grep -i "error\|fatal\|panic"

# 3. リソース確認
docker stats --no-stream
df -h
free -h

# 4. DB接続確認
docker compose exec db pg_isready -U aegis_user -d aegis_sight_db
docker compose exec db psql -U aegis_user aegis_sight_db -c "SELECT count(*) FROM pg_stat_activity;"

# 5. ネットワーク確認
docker compose exec backend curl -s http://db:5432 || echo "DB port unreachable"
docker network ls
docker network inspect aegis-sight_default

# 6. 最近の変更確認
git log --oneline -5
docker compose exec backend alembic current
```

### 4.2 障害別調査手順

#### サービス停止

| # | 手順 | コマンド |
|---|------|---------|
| 1 | コンテナ状態確認 | `docker compose ps` |
| 2 | 停止コンテナのログ | `docker compose logs <service>` |
| 3 | コンテナ再起動 | `docker compose restart <service>` |
| 4 | イメージ問題の場合リビルド | `docker compose up -d --build <service>` |

#### レスポンス遅延

| # | 手順 | コマンド |
|---|------|---------|
| 1 | リソース使用率確認 | `docker stats --no-stream` |
| 2 | スロークエリ確認 | `docker compose exec db psql -c "SELECT * FROM pg_stat_activity WHERE state = 'active' AND query_start < now() - interval '10 seconds';"` |
| 3 | 接続数確認 | `docker compose exec db psql -c "SELECT count(*) FROM pg_stat_activity;"` |
| 4 | アプリケーションログ確認 | `docker compose logs --tail=200 backend \| grep -i slow` |

#### データベース障害

| # | 手順 | コマンド |
|---|------|---------|
| 1 | DB接続確認 | `docker compose exec db pg_isready` |
| 2 | DBログ確認 | `docker compose logs --tail=100 db` |
| 3 | ロック状況確認 | `docker compose exec db psql -c "SELECT * FROM pg_locks WHERE NOT granted;"` |
| 4 | テーブル破損チェック | `docker compose exec db psql -c "SELECT schemaname, relname FROM pg_catalog.pg_statio_user_tables;"` |

## 6. コミュニケーション

```
[障害発生] AEGIS-SIGHT - P{N} 障害

```

| P2 | 30分ごと |

## 7. 障害対応訓練

### 7.1 訓練計画

| 訓練 | 頻度 | 内容 |
|------|------|------|
| 机上訓練 | 四半期ごと | 障害シナリオに基づくロールプレイ |
| 実機訓練 | 半期ごと | ステージング環境での障害対応実施 |
| カオスエンジニアリング | （将来検討） | 本番環境での意図的障害注入 |

### 7.2 訓練シナリオ例

| # | シナリオ | 期待される対応 |
|---|---------|-------------|
| 1 | DBサーバーダウン | フェイルオーバー or リストア |
| 2 | バックエンドコンテナOOMKill | リソース調整、再起動 |
| 3 | ディスク容量枯渇 | ログ削除、ストレージ拡張 |
| 4 | 認証機能障害 | ロールバック or ホットフィックス |
| 5 | ネットワーク障害 | 経路切替、DNS確認 |
